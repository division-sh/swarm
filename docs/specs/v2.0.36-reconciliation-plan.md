## 3. Phase 1 — Database Migration

**Source of truth:** `contracts/ddl-canonical.sql`  
**Output:** `migrations/023_v2_0_36_contract_alignment.sql`

### 3.1 Rebuild 5 runtime tables

| Table | Key Changes |
|-------|-------------|
| `pipeline_receipts` | `result` → `status` + `error` columns; add `ON DELETE CASCADE` on FK; add partial index on `status != 'processed'` |
| `scan_accumulators` | UUID PK → TEXT PK; `expected_agents` → `expected`; `agents_complete` → `complete`; `verticals_discovered` → `discovered`; `verticals_skipped` → `skipped`; `reports JSONB` → `reports INT`; `completed_by` default `'{}'` not `'[]'`; add `created_at`, `updated_at` |
| `pending_dedup_candidates` | `id UUID PK` → `dedup_event_id TEXT PK`; add `campaign_id`, `mode`, `name`, `signal_strength DOUBLE PRECISION`, `payload JSONB`; add `resolved_at` |
| `validation_pipelines` | `g2_spec_approved` → `g2_spec`; `g3_cto_approved` → `g3_cto`; `g4_brand_ready` → `g4_brand`; add `scoring_payload`, `packaging_requested`, `packaging_retries`; payload JSONB columns → `NOT NULL DEFAULT '{}'` |
| `template_prompt_drafts` | Add `source TEXT NOT NULL DEFAULT 'api'`, `notes TEXT`, `created_at TIMESTAMPTZ` |

### 3.2 Fix indexes

| Index | Change |
|-------|--------|
| `idx_sessions_active` | Add `COALESCE(scope_key, '')` for NULL-safe uniqueness |
| `idx_conversations_scope` | Add `COALESCE(scope_key, '')`, broaden WHERE to `status = 'active'` (not just `session_per_vertical`) |

### 3.3 Fix runtime_config table

Remove `config_hash`, `applied_by` (not in migration 003). Add `config_path TEXT NOT NULL`, `created_at TIMESTAMPTZ NOT NULL DEFAULT now()`.

### 3.4 Migration strategy

For runtime tables that are stateful (scan_accumulators, validation_pipelines), the migration must preserve in-flight data. Recommended approach: `ALTER TABLE ... RENAME COLUMN` where possible, `ADD COLUMN` with defaults for new columns. For pending_dedup_candidates (PK type change), if the table has no live rows, `DROP TABLE` and recreate is simplest.

## 10. Risk Mitigations

### 10.1 haiku_or_sonnet model tier

Before setting `haiku_or_sonnet` on chief-of-staff, vp-product, and vp-growth, verify the model tier resolution path. Check agent spawner code for an exhaustive tier switch — if it only handles `haiku` and `sonnet`, add a resolution case for `haiku_or_sonnet` that selects based on task complexity or defaults to `sonnet`.

### 10.2 agent_message universality

The contract says `agent_message` is a universal tool (injected into ALL agents). Before removing it from per-agent tool lists in config YAMLs, verify `extractAllowedTools()` in `tool_executor.go` injects universal tools independently of the config list. If the config list is treated as the complete tool set, removing `agent_message` will break inter-agent communication.

### 10.3 Running verticals

If there are live OpCo verticals with data in `routing_rules`, the migration should clean routing entries for ghost events. Add to migration:
```sql
DELETE FROM routing_rules WHERE event_pattern IN (
  'mandate_updated', 'launch_ready', 'channel_update', 'market_feedback'
);
```

### 10.4 Leadership route removal safety

The most dangerous change in Phase 3 is removing leadership-targeted routes from routes.yaml. If the EventBus static subscription path doesn't deliver OpCo-scoped events to leadership agents, removing the routing_rules fallback will silently break CEO/CoS/VP event delivery. Recommended: verify static delivery works in a test vertical first. Ship route removal as a separate, revertable PR.

### 10.5 System prompts

DO NOT modify `system_prompt` fields in any config YAML during this reconciliation. Prompts are outside contract scope and are owned by the prompt engineering process, not the spec.

### 10.6 customer_message normalization

The contract documents a known misalignment: `support-agent` config subscribes to `customer_message` but `inbound.go` emits `inbound.{slug}.whatsapp_message` / `inbound.{slug}.email`. This reconciliation does NOT fix this — it requires a design decision on whether inbound.go should translate to `customer_message` or support config should subscribe to `inbound.*` events. Track as a follow-up.
